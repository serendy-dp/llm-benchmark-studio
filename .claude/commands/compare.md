# /compare [ベンチマークファイル名] — モデル比較評価レポート

指定ベンチマークの採点済み結果を全モデル分読み込み、スコア・実行時間・モデル特性を多角的に分析して比較レポートを生成する。

## 実行手順

### 1. 対象の特定

**引数あり:** `results/scored/` から `meta.benchmark_file` が引数と一致するファイルを全件収集。

**引数なし:** `results/scored/` にある全JSONの `meta.benchmark_file` の種類を列挙し、ユーザーに選択を求める。

対象が1件以下（比較不可能）の場合はその旨を報告して終了。

### 2. ベンチマーク定義の読み込み

`benchmarks/<benchmark_file>` を読み込み、以下を把握する：
- `benchmark_meta.title` / `description`：ベンチマークの目的・評価領域
- 各ドメインの `label` / `description`：評価観点の意味
- 問題ごとの `difficulty` と `tags`：難易度分布・評価能力の種類

### 3. 結果データの集計

各結果ファイルから以下を抽出する：

#### スコア指標
- 全体平均スコア（スコア付き問題の平均、小数第2位）
- ドメイン別平均スコア
- スコア分布：5/3/1/0 の問題数と割合
- 満点（5）問題の割合：高いほど「確実性が高い」モデル
- 0点問題の割合：高いほど「致命的誤りが多い」モデル

#### 実行時間指標
- 全問題の平均・中央値・最大 `elapsed_sec`
- ドメイン別の平均応答時間
- 最初の問題の elapsed_sec は参考値（ウォームアップ済みか不明な場合あり）

#### 採点コメントの傾向
- `score_comment` が記録されている問題の内容を横断して読み取り、各モデルの共通した失敗パターンや特徴を言語化する

### 4. モデル特性の分析

各モデルについて以下を評価する：

**得意領域：** スコア 5 が多いドメイン・問題タグ。どのような能力に長けているか。

**弱点領域：** スコア 0/1 が集中するドメイン・問題タグ。どのような失敗が起きているか（score_comment や response の傾向から言語化）。

**速度特性：** 平均応答時間を以下の3段階で分類：
- 高速（< 5秒平均）
- 中速（5〜15秒平均）
- 低速（15秒超）

**出力制御精度：** 「URLのみ」「数字のみ」のような指示に対し、余分な出力を付加するかどうか。スコア3が多い場合は出力制御が課題と判断。

### 5. 総合評価

ベンチマークの `description` と各ドメインの `description` をもとに、この評価が想定する**実用場面**（例: Webスクレイピングエージェント、自動フォーム検出パイプライン等）を明記したうえで：

1. **精度優先シナリオ**（ミスが許されない本番運用）に最適なモデルとその理由
2. **速度優先シナリオ**（大量URLの一括処理）に最適なモデルとその理由
3. **バランスシナリオ**（精度と速度のトレードオフを許容）に最適なモデルとその理由
4. **総合推奨モデル**：平均スコア・速度・失敗パターンのリスクを総合して最優秀を選定し、理由を2〜3文で説明

### 6. レポートの出力

以下の構造でMarkdownレポートを出力する：

```
# モデル比較評価レポート — <benchmark_title>
**生成日:** <YYYY-MM-DD>  **ベンチマーク:** <benchmark_file>

---

| モデル | 全体平均 | 平均応答時間 | 満点率 | 0点率 |
|--------|---------|------------|--------|-------|
| ...    |         |            |        |       |

（平均スコア降順でソート）

---

## ドメイン別スコア比較

| ドメイン | <モデル1> | <モデル2> | ... |
|---------|-----------|-----------|-----|

---

**出力制御：** 良好/課題あり
**代表的な失敗パターン：** ...（score_comment・responseから）

---

### このベンチマークが想定する実用場面
...

### ユースケース別推奨

| 精度優先 | | |

>

```

### 7. ファイルへの保存

レポートを以下のパスに保存する：

```
results/reports/<benchmark_stemname>_compare_<YYYYMMDD>.md
```

ディレクトリが存在しない場合は作成する：
```bash
mkdir -p results/reports/
```

その後、保存先パスをユーザーに報告する。
